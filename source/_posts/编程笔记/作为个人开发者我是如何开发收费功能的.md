这篇文章想分享下作为个人开发者，我是如何开发一个收费插件的？

## 为什么要做收费功能？

这个插件最开始只是我给自己写的工具。因为思源笔记很适合组织资料和笔记，但是原生在任务管理方面支持不够，所以我才开始动手开发这个插件，提供任务提醒、项目管理、番茄钟计时等功能。

后来插件用的人变多了，需求也越来越多。有人希望移动端也能通知，有人希望习惯、番茄钟、任务能联动起来……插件从一个“我自己能用就行”的工具，慢慢变成了一个需要长期维护的项目。

这时候就会遇到一个很现实的问题：时间精力有限，如果没有任何收益，我只会优先满足自己的需求，修自己遇到的问题，做自己想要的功能。其他用户的需求，如果和我的使用场景不一致，我是不会去优化的。

一开始是选择让用户给我打赏，我来满足他的需求，但是后面就发现这个方案很难持续。

主要有以下问题

1. 因为大部分用户基本是为了让我加功能，才给我打赏，而不是因为现有功能很满意给我打赏，收到打赏就意味着我要满足他的需求，开发新功能，压力比较大。
2. 用户肯给我打赏50其实已经算多了，但是对我而言，50块钱要花费我一两个小时甚至更多的时间，这个钱对我而言是少的。用户因为一个功能给我打赏后，又有了新的需求，又给我打赏，这样可能会导致一个用户反复打赏，ta花的钱过多（比如超过几百），我拿到的又相对较少，对于双方而言，都是一个负担。
3. 插件容易因为满足少数几位打赏用户的需求，而无法掌控我自己的功能发展方向。

所以我后来决定给插件加付费功能。这样可以让开发成本平摊到每个付费用户身上，有功能需求的用户花的钱少了，我也能得到更多的收益，并且更有动力去维护和改进插件，以及补充使用文档。

而对于个人开发者而言，做付费功能，最难的不是写代码，而是解决怎么收钱。

为了给插件加付费功能，我也做了挺久的调研，尝试了几种方案，最后才确定了现在的方案。这里简单分享下我的经验。

## 会员功能激活码如何设计？

设计是用户通过我发放的「**激活码**」来激活会员功能。

激活码要支持离线验证。因为思源笔记有很多离线用户。有些用户的笔记库在内网环境，有些用户习惯长期断网使用。

我的做法是使用非对称加密签名：

- 我这边保存私钥，用私钥生成激活码
- 插件前端只内置公钥，用公钥验证激活码

这样可以实现

- 私钥不出现在插件代码里，用户无法伪造激活码
- 公钥即使暴露也没关系，因为公钥只能验证，不能生成

激活码不是一个随便生成的随机字符串，而是把思源笔记账户 ID、购买时间、订阅期限一起参与签名。大概逻辑是：

```text
message = 用户ID + 购买时间 + 订阅期限
signature = 私钥签名(message)
activation_code = 购买时间编码 + 订阅期限 + signature
```

之所以绑定思源笔记账户 ID，是因为思源账户 ID 是唯一的。这样同一个激活码只能给对应用户使用，别人复制过去也无法通过验证。而对于离线用户而言，只需要在有网的电脑端登录思源账号，把思源的conf文件拷贝进离线电脑，就能实现登录，这样离线电脑也能使用上激活码

添加了购买时间编码 + 订阅期限，这样就可以通过购买时间+订阅期限本地推算出过期时间。

## 付费流程是手动还是自动化？

最简单的方式当然是：直接把微信收款码和支付宝收款码放出来。

用户扫码付款，然后加我微信或者 QQ，把用户 ID 发给我。我确认收款之后，再手动生成激活码发给用户。

这个方案从开发成本而言是最省事的，但后续的时间成本却是很高的，意味着每次有用户购买，都需要花时间去处理。

因为我还是研究生，平时有自己的学习、实验和工作安排。我不希望自己正在工作的时候，总是被“我已经付款了，麻烦发一下激活码”打断。从用户角度，如果用户中午付费，而我晚上才看到消息，他就要等我半天才能用上功能，那么对用户来说，这个体验也不好。

所以我一开始就确定了一个目标：付费流程要自动化。

用户应该可以在插件里直接扫码付款，付款后插件自动查询订单状态，确认支付成功后自动拿到激活码。整个过程不需要联系我，也不需要等我手动处理。

## 支付平台怎么选

如果直接接支付宝、微信官方支付平台的API，是需要营业执照的。对于个人开发者来说，这一步比较麻烦。

所以我选择的是第四方支付平台Zpay。

> 链接：[https://z-pay.cn/?uid=23304](https://z-pay.cn/?uid=23304)

选择它的主要原因：

- **API功能完善**：支持生成付款二维码、查询订单是否支付等功能
- **可以直接走支付宝平台**：支付宝v1接口走的是支付宝官方的当面付，用户付款后，我能直接在支付宝看到收款信息，不需要在平台手动提现，不怕平台跑路

  ![64d0068567927fa65970bfe139dfd1d4](https://fastly.jsdelivr.net/gh/Achuan-2/PicBed@pic/assets/64d0068567927fa65970bfe139dfd1d4-20260520151037-ok8gd3h.jpg)
- **抽成低**：我使用的支付宝 v1 接口开户费是 88 元，收款抽成大概是 1.0% 平台手续费 + 0.6% 支付宝官方手续费。这个比例对我来说可以接受。相比之下，有些平台抽成能到 3-6%。

## 为什么还需要一个服务端

选好支付平台之后，还有一个问题：这些支付逻辑放在哪里？

因为我写的是思源笔记插件，本质上是前端代码。如果我直接在插件里调用支付平台接口，或者在插件里生成激活码，那么支付密钥、签名逻辑、激活码生成逻辑都会暴露出来。

这显然不合适。

付费功能里真正敏感的部分必须放到服务端：

- 支付平台密钥不能放在插件里
- 激活码公钥可以放在插件里，但私钥不能放在插件里
- 订单是否支付成功不能只相信前端
- 激活码应该由服务端生成，插件只负责验证

因此，最终的结构是：

```text
思源插件前端
  -> 通过调用API请求服务器
  -> 服务器调用支付平台
  -> 服务器读写 D1 数据库
  -> 服务器返回二维码、订单状态、激活码
```

插件只做展示和本地保存，真正的支付逻辑放在服务器上。

服务端，我选择的是 Cloudflare Worker。原因也很简单：

- **免费够用**：Vercel虽然有免费的Hobby计划，但仅限个人非商业用途，不能开发付费应用；而Cloudflare Worker没有这个限制，并且免费额度是每天 10 万次请求，对于个人开发者来说已经非常充足了。
- **没有被墙**：除了福建等少数地区外，国内大部分地区是可以直接访问Cloudflare Worker，这样用户在使用插件时就不需要科学上网了。
- **方便和数据库联动**：Cloudflare Worker可以直接绑定D1数据库和R2对象存储，这样就可以存储订单、激活码和用户订阅状态等信息。

## Cloudflare Worker + D1 具体部署过程

### 整体思路

我的付费服务大概分成几类接口：

- `/api/create-payment`：创建支付订单，返回二维码
- `/api/check-status`：查询订单状态，支付成功后返回激活码
- `/api/subscription`：查询用户订阅状态和已有激活码
- `/api/freeTrialUsed`：查询用户是否领取过试用

数据库里主要存四类数据：

- `orders`：订单记录
- `activation_codes`：激活码
- `subscriptions`：订阅状态
- `free_trial_used`：试用记录

```text
用户点击购买
  -> Worker 创建订单
  -> 支付平台生成二维码
  -> 用户扫码付款
  -> Worker 查询支付状态
  -> 支付成功后生成激活码
  -> 插件本地验证并保存激活码
```

### 第一步：准备 Cloudflare 账号和 Wrangler

先注册 Cloudflare 账号，免费版即可：

[https://dash.cloudflare.com/sign-up](https://dash.cloudflare.com/sign-up)

本地需要安装 Node.js 和 Wrangler。Wrangler 是 Cloudflare 的命令行工具，用来创建 D1、运行 Worker、部署服务和查看日志。

```bash
# 如果没安装 Node.js，先去 https://nodejs.org/ 下载安装

# 安装 Wrangler
npm install -g wrangler

# 登录 Cloudflare
wrangler login
```

登录时浏览器会打开授权页面，授权完成后，命令行就可以操作 Cloudflare 账号下的 Worker 和 D1 了。

### 第二步：创建 Worker 项目

我把支付服务单独放在一个项目里，不和插件前端混在一起。

项目结构大概是：

```text
worker
├─ src
│  └─ index.js
├─ schema.sql
├─ wrangler.toml
├─ package.json
└─ package-lock.json
```

### 第三步：创建 D1 数据库

D1 是 Cloudflare 提供的 SQLite 数据库。对这个付费系统来说，它主要用来保存订单、激活码和订阅状态。

创建数据库：

```bash
wrangler d1 create 数据库名称
```

命令执行后终端会返回 `database_id`​，这个值后面要写到 `wrangler.toml` 里。

### 第四步：创建数据库表结构

创建 `schema.sql`。

```sql
-- 订单表
CREATE TABLE IF NOT EXISTS orders (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    out_trade_no TEXT UNIQUE NOT NULL,
    user_id TEXT NOT NULL,
    product_name TEXT NOT NULL,
    amount TEXT NOT NULL,
    status INTEGER DEFAULT 0,
    term TEXT NOT NULL,
    trade_no TEXT,
    created_at TEXT NOT NULL,
    paid_at TEXT
);

-- 激活码表
CREATE TABLE IF NOT EXISTS activation_codes (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    code TEXT UNIQUE NOT NULL,
    user_id TEXT NOT NULL,
    term TEXT NOT NULL,
    order_id INTEGER,
    created_at TEXT NOT NULL,
    expires_at TEXT,
    FOREIGN KEY (order_id) REFERENCES orders(id)
);

-- 用户订阅表
CREATE TABLE IF NOT EXISTS subscriptions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id TEXT UNIQUE NOT NULL,
    expire_date TEXT NOT NULL,
    is_lifetime INTEGER DEFAULT 0,
    updated_at TEXT NOT NULL
);

-- 创建索引
CREATE INDEX IF NOT EXISTS idx_orders_user_id ON orders(user_id);
CREATE INDEX IF NOT EXISTS idx_orders_out_trade_no ON orders(out_trade_no);
CREATE INDEX IF NOT EXISTS idx_activation_codes_code ON activation_codes(code);
CREATE INDEX IF NOT EXISTS idx_activation_codes_user_id ON activation_codes(user_id);
CREATE INDEX IF NOT EXISTS idx_subscriptions_user_id ON subscriptions(user_id);

-- 试用记录表
CREATE TABLE IF NOT EXISTS free_trial_used (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id TEXT UNIQUE NOT NULL,
    created_at TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_free_trial_used_user_id ON free_trial_used(user_id);
```

初始化本地数据库：

```bash
wrangler d1 execute 数据库名称 --local --file=./schema.sql
```

初始化远程数据库：

```bash
wrangler d1 execute 数据库名称 --remote --file=./schema.sql
```

验证表是否创建成功：

```bash
# 查看本地数据库的表
wrangler d1 execute 数据库名称 --local --command="SELECT name FROM sqlite_master WHERE type='table'"

# 查看远程数据库的表
wrangler d1 execute 数据库名称 --remote --command="SELECT name FROM sqlite_master WHERE type='table'"
```

验证索引：

```bash
# 查看本地数据库的索引
wrangler d1 execute 数据库名称 --local --command="SELECT name FROM sqlite_master WHERE type='index'"

# 查看远程数据库的索引
wrangler d1 execute 数据库名称 --remote --command="SELECT name FROM sqlite_master WHERE type='index'"
```

### 第五步：配置 wrangler.toml

`wrangler.toml` 是 Worker 的配置文件，里面写 Worker 入口、D1 绑定、环境变量和自定义域名。

我的配置大概是这样：

```toml
name = "worker名称"
main = "src/index.js"
compatibility_date = "2024-01-01"

[[d1_databases]]
binding = "DB"
database_name = "数据库名称"
database_id = "你的 database_id"

[vars]
PID = "你的 zpay 商户 ID"
ZPAY_API_BASE = "https://zpayz.cn"
```

### 第六步：配置密钥

为了安全起见，支付平台的密钥和激活码私钥不要写进代码，也不要写进 `wrangler.toml`。

用 Wrangler 的 secret：

```bash
# zpay 密钥
wrangler secret put PKEY

# 生成激活码用的 ECC 私钥
wrangler secret put PRIVATE_KEY

# 管理接口 token，用来手动生成激活码
wrangler secret put ADMIN_TOKEN
```

输入命令后，终端会让你输入具体值。这个值会保存到 Cloudflare，不会出现在仓库里。

之后部署后，可以在设置的 Variables and Secrets 进行管理

![PixPin_2026-05-20_13-00-20](https://fastly.jsdelivr.net/gh/Achuan-2/PicBed@pic/assets/PixPin_2026-05-20_13-00-20-20260520130036-6sc91wi.png)

### 第七步：编写 Worker 接口

Worker 的入口是 `src/index.js`。

主要功能

- 创建订单
- 查询订单并生成激活码，如果用户选择的是免费试用的话，则直接发放激活码，限制一个用户只能试用一次。
- 查询特定用户的所有激活码
- 给指定用户发放激活码

### 第八步：本地测试和部署

本地测试：

```bash
wrangler dev --remote
```

部署到生产环境：

```bash
wrangler deploy
```

部署成功后，会得到一个 Worker 地址，类似：

```text
https://siyuan-tasknote-payment.xxx.workers.dev
```

之后修改代码，只需要运行`wrangler deploy`即可

### 第九步：绑定自定义域名

Worker 默认域名可以直接用，但我还是绑定了自己的域名

由于之前的域名是在阿里云买的，DNS解析用的是阿里云，CloudFlare Worker的自定义域名要求域名**需要托管于 CloudFlare**上，所以需要迁移下

可参考：[详细图文手把手教你阿里云注册域名如何托管到CloudFlare DNS服务](https://zhuanlan.zhihu.com/p/703861587)

之后在Woker设置路由关联，修改域名

`wrangler.toml` 里配置：

```toml
routes = [
  { zone_name = "域名", pattern = "项目子域名", custom_domain = true }
]
```

### 第十步：插件添加功能调用API

Worker 部署完成后，插件前端只需要调用公开 API，实现创建支付、查询订单并生成激活码即可。
